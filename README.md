# EM Testcase Auth App

## Описание проекта

Backend-приложение, реализующее собственную систему аутентификации и авторизации.

Проект выполнен как тестовое задание и демонстрирует:
- ручную реализацию auth-логики без использования готовых решений фреймворка
- собственную систему управления пользователями
- базовую модель разграничения доступа (RBAC-подобная логика)
- контроль доступа к ресурсам на уровне бизнес-логики

---

##  Технологии

- Python 3.x
- FastAPI
- SQLAlchemy (ORM)

## Основной функционал

### 1. Работа с пользователями

Реализованы базовые операции:

- регистрация пользователя
- получение списка пользователей
- обновление данных пользователя
- удаление пользователя (soft delete через флаг активности)

---

### 2. Аутентификация (без JWT / без встроенных механизмов)

В проекте реализована **собственная простая сессионная модель**:

- вход пользователя осуществляется через email + password
- пользователь идентифицируется на последующих запросах через механизм хранения состояния (session / middleware / dependency)
- доступ к ресурсам определяется на серверной стороне

---

### 3. Авторизация (система доступа)

Реализована простая система разграничения доступа.

#### Основная идея:

Доступ к ресурсам определяется через связку:

- пользователь → роль → разрешения

#### Сущности:

- User
- Role
- Permission

#### Логика:

- пользователь имеет одну роль
- каждая роль содержит набор разрешений
- разрешения определяют доступ к операциям над ресурсами

---

### 4. Обработка ошибок доступа

Реализована стандартная модель HTTP-ответов:

- `401 Unauthorized` — пользователь не определён (не выполнен login)
- `403 Forbidden` — пользователь определён, но не имеет прав

---

### 5. Административная логика

Реализованы базовые возможности управления системой:

- назначение ролей пользователям
- управление разрешениями
- настройка доступа к ресурсам

(реализация упрощённая, через API / сервисный слой)

---

### 6. Mock бизнес-ресурсы

Для демонстрации системы авторизации используются mock-объекты:

- условные "ресурсы приложения"
- операции чтения / изменения
- доступ регулируется системой permissions

## Как работает система

1. Пользователь регистрируется
2. Выполняет login
3. Пользователь идентифицируется в системе через middleware / dependency
4. При каждом запросе:
   - определяется пользователь
   - проверяются его роли и permissions
5. Выдаётся результат:
   - доступ разрешён
   - или ошибка 403

## Запуск проекта
Проект использует dependency manager **uv**.

### Установка uv
```bash
pip install uv
```

### Клонирование
```bash
git clone https://github.com/ChaRoodey/em_testcase_auth_app
cd em_testcase_auth_app
```

### Перед запуском проекта создайте файл `.env` на основе примера:

```bash
cp .env.example .env
```

### Установка и запуск

```bash
uv sync

uv run alembic upgrade head

uv run uvicorn app.main:app --reload
```
